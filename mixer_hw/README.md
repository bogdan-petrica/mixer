# Introduction

This document is presenting the hardware part of an hardware/software audio mixer project.

More information about the project can be seen [here](../README.md).

Many of the [goals](../README.md#design-goals) are achieved by using a dedicated __audio mixer IP Core__.

Most of this document is concerned with the __audio mixer IP Core__ and the integration to the platform project.

The document targets technical reviewers and engineers.

# Target hardware

Target device is [__Zynq 7000__](https://docs.amd.com/r/en-US/ug585-zynq-7000-SoC-TRM/Introduction?tocId=Hf6C7Oo5ABvv2hkWRoiihQ) and the target board is Digilent [Zybo Z7-7010](https://digilent.com/reference/programmable-logic/zybo-z7/start).

The audio codec used by __Zybo Z7__ is [SSM2603](https://www.analog.com/media/en/technical-documentation/data-sheets/SSM2603.pdf), provided by Analog Devices.

# Used software

The project was developed using [AMD Vivado 2024.2](https://www.xilinx.com/support/download/index.html/content/xilinx/en/downloadNav/vivado-design-tools/2024-2.html).

# Goals and constraints

## Platform goals

Besides the __audio mixer IP Core__ goals the platform should aim to achieve the following goals:
- control over the __SSM2603__ audio codec using the __I2C__ protocol, the __SSM2603__ is directly connected to the __Zybo Z7-7010__ FPGA IO ports, see [audio](https://digilent.com/reference/programmable-logic/zybo-z7/reference-manual#audio) section on Digilent website
- provide access to __Zybo Z7-7010__ buttons, switches and LEDs for better UX
- reuse of available IP Cores, the DMA for samples transfer to and from the __audio mixer IP Core__ can be realized with [__AXI DMA IP Core__](https://docs.amd.com/r/en-US/pg021_axi_dma/), the communication with __SSM2603__ over __I2C__ can be done with the help of [__IIC IP Core__](https://docs.amd.com/v/u/2.0-English/pg090-axi-iic)

## __Audio mixer IP Core__ goals and constraints

### Goals
- independent control plane, accessible over __AXI4 LITE__
- __I2S__ playback and recording data plane
- channel gain control
- multi channel audio mixing with saturation
- DMA Device to Memory (recording)
- DMA Memory to Device (playback)
- PS buffered recording and buffered playback - for relaxed realtime requirements on the host CPU
- configurable __delay line__

### Other goals
- simplicity
- code auditability over raw performance
- high test coverage trough test benches - both functional and integration test benches for non trivial cases

### Non goals
- generic and reusable implementations

# Getting started with the project - Workflow

All the sources, block diagrams, constraint files and block diagram HDL wrappers used by Vivado are version controlled within this repository. The process of working with Vivado using this project repository is described in this section.

The supplied [import_project.tcl](./import_project.tcl) is used to create the Vivado project.

To create the project, open Vivado, go to the TCL console and type the following commands:

```tcl
# go to top level directory
cd mixer_hw

# call the creation script
source import_project.tcl
```

The Vivado project is configured to reference the files in the repository, all the VHDL sources, block diagrams and constraint files are referenced, saves directly to the repos files.

The only generated file by Vivado tracked within this repository is the `./bd/hdl/design_1_wrapper.vhd`,which is automatically managed. This happens because Vivado both generates and references the HDL wrapper. The approach has no other drawback other than each time the VHD wrapper is regenerated, it needs to be committed.

All files under `./bd/` directory except `./bd/design_1.bd` and `./bd/hdl/design_1_wrapper.vhd` are generated by Vivado and can be deleted.

The [import_project.tcl](./import_project.tcl) creates the Vivado project under `build` directory, the directory is wiped out before creating the project.

__Note:__ Vivado also generates files under `./bd/` directory which are not under version control, these are ignored by version control but not deleted.

__Note:__ If you change the project such that multiple block diagrams and HDL wrappers are used, the workflow has to be adapted to account for the new files.

Once you have the Vivado project you can start doing the required changes.

Once satisfied with the changes to the files, check that:
- required test benches are added/modified
- all test benches pass
- timing and utilization reports show appropriate values for synthesized and implemented designs
- synthesis and implementation finishes successful

When all the above are done, you can go and export the project settings by going to the Vivado TCL in the top level directory and call the `export_project.tcl` script, as follows:

```tcl
# go to top level directory of the repo
cd mixer_hw

# call the export script
source ./export_project.tcl
```

After the export is successful you are ready to commit the changes.

# IP Cores used

The following IP cores are third party sourced:
- [AMD PS7](https://docs.amd.com/v/u/5.3-English/pg082-processing-system7) v5.5 - the __Zynq PS__
- [AXI IIC](https://docs.amd.com/v/u/2.0-English/pg090-axi-iic) v2.1 - for communication with __SSM2603__ over __I2C__ protocol
- [AXI DMA](https://docs.amd.com/r/en-US/pg021_axi_dma) v7.1 - for DMA between __audio mixer IP Core__ and the __Zynq PS__
- [AXI GPIO](https://docs.amd.com/r/en-US/pg144-axi-gpio) v2.0 - for reading __Zybo Z7-7010__ button and switches
- [PROC RST](https://docs.amd.com/v/u/en-US/pg164-proc-sys-reset) v5.0 - provides reset signals for peripherals and bus controllers
- [AXI SMC](https://www.xilinx.com/support/documents/ip_documentation/smartconnect/v1_0/pg247-smartconnect.pdf) v1.0 - AXI bus controller, connects devices over AXI and provides AXI protocol conversions
- [CLOCKING WIZARD](https://docs.amd.com/r/en-US/pg065-clk-wiz) v6.0 - for generating the __12.288 MHZ__ clock  used by the audio codec
- [SSM2603 Audio Codec](https://www.analog.com/media/en/technical-documentation/data-sheets/SSM2603.pdf) - provided by Analog devices and present on the __Zybo Z7-7010__ board

The __Audio Mixer IP Core__ documentation can be checked [here](./doc/AudioMixerIP.md).

# Utilization and timing reports

## Clocks

There are two clocks used in the design:
- __50 MHZ__ for AXI bus interface and control logic, the __AXI4 LITE__ and __AXI Stream__ IP cores use these bus interfaces for connectivity with the system host CPU
- __12.288 MHZ__ for audio codec __I2S__ interface and mixing logic, as required by [SSM2603](https://www.analog.com/media/en/technical-documentation/data-sheets/SSM2603.pdf)
